Arrays & Hashes

From the basics to differences in compute time

Arrays and hashes are both fundamentally important because they help us control, keep track of, and access great big bundles of data. The Ruby Docs tell us that an array is “an ordered, integer-indexed collection of any object.” On the other hand, a hash “is a dictionary-like collection of unique keys and their values.” So, they’re quite similar. The main difference boils down to this- you can use any object as an index of a hash, but an array is automatically indexed with integers.

Given the array below, you could access "pizza" if you knew it's index value. With the hash, you could access the "pizza"s value without knowing precisely where it is in the hash. As a bonus, it would give you even further information about that pizza.

									
junk_food_array = [“popcorn”, “pizza”, “cake”]
junk_food_array[1]
#Output:
#pizza

junk_food_hash = { popcorn: “snack”, pizza: “dinner”, cake: ““dessert””}
junk_food_hash[:pizza]
#Output:
#dessert

Of course, both hashes and arrays have unique qualities that make them a better or worse choice in any given situation. However, it was only recently that I started to consider the cost in compute time when using an array instead of a hash.

If we recall from my last blog post about Big O Notation, the best possible runtime is O(1) or constant time. That means the compute time will be the same no matter what the input size is. Amazingly, the lookup time for a hash is O(1). The lookup time for an array (using .find) is O(n), which is still fine, but not as good.

Let's consider the junk food example above one more time. What if someone else had all the data about junk food, and there were thousand and thousands of junk food entries recorded. Our job might be to check this list to see if a certian food item is considered junk food. In this case, there would be no way to know the index value of certain food item, we wouldn't even be sure if it was there at all. We would have to use the .find enumerable.

									
junk_food_array.find {|i| i == "pizza"}
#Output:
#pizza

The .find enumerable has to go through each item in the array before it reaches the end or comes across a matching "pizza" entry. Meanwhile, because of the way hash tables are built on the backend, junk_food_hash[:pizza] just plucks the pizza key's value of "dinner" right out. There's no need to sort through each key/value pair, and that's kind of the point of hashes.

For more information about how hashes store information on the backend so efficiently, Gayle Laakmann McDowell gives a great description on HackerRank's YouTube channel: Data Structures: Hash Tables.
